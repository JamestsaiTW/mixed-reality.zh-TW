# <a name="comfort"></a>緩和

在自然的檢視，人力 visual 系統依賴多個來源的詳細資訊，或"提示，，"來解譯 3D 圖形和物件的相對位置。 一些提示只依賴單一眼睛 （monocular 提示），包括[線性觀點來看](https://en.wikipedia.org/wiki/Perspective_(graphical))，[熟悉大小](https://en.wikipedia.org/wiki/Size#Perception_of_size)，阻擋，[深度的欄位模糊](https://en.wikipedia.org/wiki/Depth_of_field)，和[住宿](https://en.wikipedia.org/wiki/Accommodation_(eye))。 其他提示依賴這兩個眼睛 （雙眼提示），並包含[vergence](https://en.wikipedia.org/wiki/Vergence) （基本上，才能查看物件的眼睛相對旋轉） 和[雙眼差異](https://en.wikipedia.org/wiki/Stereopsis)（模式的差異之間的背面的兩個眼睛場景投影）。 若要確保在 head 掛接的顯示器上最大緩和，很重要的設計人員和開發人員建立，並模擬這些提示在自然的世界中的運作方式的方式呈現內容。 實體的觀點而言，也很重要的設計不需要的頸部或手臂 fatiguing 動作的內容。 在本文中，我們將逐一要牢記在心，以達成這些目標的重要考量。

## <a name="vergence-accommodation-conflict"></a>Vergence 住宿衝突

若要清楚地檢視物件，必須讓人判讀[容納](https://en.wikipedia.org/wiki/Accommodation_%28eye%29)，或調整其眼睛的焦點，物件的距離。 在此同時，這兩個眼睛的旋轉角度必須[聚合](https://en.wikipedia.org/wiki/Convergence_(eye))物件的距離，以避免看到 double 的映像。 在自然的檢視，vergence 和住宿連結。 當您檢視項目 (例如接近您鼻子 housefly) 附近眼睛跨，並配合 near 點。 相反地，如果您在光學無限大 （大約會開始在 6 分鐘或更一般的願景） 檢視的項目，您的眼睛的視線成為平行和眼睛的功能濾鏡配合，為無限大。 

在最前端掛接顯示使用者一律可容納焦點距離 （以取得明確的映像） 的顯示，但同時回歸到距離 （以取得單一映像） 感興趣的物件。 當使用者配合，並同時回歸到不同的距離時，自然連結之間的兩個提示必須中斷，這可能會導致 visual discomfort 或疲勞。

### <a name="best-practices-for-hololens-1st-gen-and-hololens-2"></a>HoloLens 的最佳作法 （第 1 代） 和 HoloLens 2

最大的緩和**全像放置的最佳區域是之間 1.25 m 和 5m**。 每個案例中，在設計工具應該嘗試鼓勵使用者互動的結構內容場景 1 百萬個或與內容較遠 (例如調整[內容大小，並在預設位置參數](gaze-targeting.md))。 

雖然內容可能偶爾會需要更接近於 1 分鐘顯示，但我們建議您不要曾經呈現更接近比 40 cm 全像投影中。 因此，我們建議開始**淡出 40 cm 的內容，並放在轉譯裁剪平面 30 cm**以避免任何比較接近物件。

深入了解移動的物件都是靜態物件，以產生 discomfort 由於 vergence 住宿衝突，導致更有可能的。 同樣地，需要快速切換接近焦點和最焦點 （例如，因為快顯的雷射，需要直接互動） 的使用者可能會導致 visual discomfort 和疲勞。 因此，**額外小心頻率降到最低的使用者： 檢視內容，深入了解; 移動或快速之間切換焦點遠近全像投影**。 

### <a name="additional-considerations-for-hololens-2-and-near-interaction-distances"></a>HoloLens 2 和互動距離附近的其他考量

設計 (near) 互動 HoloLens 2 中的直接內容時或**任何應用程式中的內容必須放置的位置更接近於 1 分鐘，額外小心以確保使用者緩和**。 Discomfort 由於 vergence 住宿衝突的可能性以指數方式增加與減少檢視距離。 此外，使用者可能會遇到增加的 bluriness 互動距離近檢視的內容，因此我們建議您測試的內容呈現最佳的全像放置也以更接近 (小於 1.0 m 裁剪平面下) 的區域內時要請確定它會維持清楚且方便檢視。 

**我們建議您建立應用程式的使用者應該在檢視即將 (小於 1.0 m) 的內容的時間量為基礎，並深入了解移動的 「 深度 budget"** 。 舉例來說，若要避免將使用者放在這些情況下超過 25%的時間。 如果超過深度預算時，建議您小心使用者測試以確保它保持舒適的體驗。 

一般情況下，我們也建議仔細測試，以確保在任何互動需求 （例如，下列項目的速度移動、 連線能力等），互動距離近保持舒適的使用者。 


### <a name="guidance-for-immersive-devices"></a>沈浸式裝置的指引

沈浸式裝置，指引和最佳作法 HoloLens 仍然適用，但為區域的緩和的特定值移位根據焦點距離，以顯示。 一般情況下，這些會顯示焦點不平均的距離是之間 1.25 m-2.5 m。 如有疑問，避免太接近使用者感興趣的物件，並改為嘗試讓大部分的內容 1 百萬個的轉譯或遠處。

## <a name="interpupillary-distance-and-vertical-offset"></a>Interpupillary 距離和垂直位移

當檢視數位內容標頭上掛接，會顯示 (HMD)，相對於顯示位置的數位內容檢視器的眼睛位置很重要。 具體來說，這兩個 interpupillary 的距離 ([IPD](https://en.wikipedia.org/wiki/Pupillary_distance)) 而且重要舒適的檢視中 HMDs 的數位內容的垂直位移 (VO)。 

IPD 指 pupils 或個人的眼睛的中心之間的距離。 VO 指的是潛在的垂直位移的數位眼睛相對於檢視器的眼睛的水平軸顯示的內容 （值得注意的是，這是不相同的水平位移，或雙眼差異）。 錯誤比對一個或兩個個別使用者到這些因素可以降低 discomfort vergence 住宿衝突所造成的影響，但它可能會在即使原因 discomfort V 的衝突降到最低 （例如，針對主要 2.0 m 在顯示的內容時距離的 HoloLens）。 

### <a name="guidance-for-holographic-devices"></a>適用於全像攝影版裝置的指引

HoloLens 的 （第 1 代），預估 IPD 且在裝置設定[校正](calibration.md)。 已設定新使用者的裝置，您必須執行校正或 IPD 必須手動設定。 VO 完全取決於適合的裝置。 具體來說，VO 降到最低，裝置必須將滑鼠停在使用者的大腦，因此顯示會使用他/她的眼睛的軸的層級。 

#### <a name="hololens-2"></a>HoloLens 2

HoloLens 2 IPD 是估計和期間眼睛/裝置設定[校正](calibration.md)。 已設定新使用者的裝置，校正必須執行才能確定 IPD 已正確設定。 VO 會佔用自動 HoloLens 2 中。

## <a name="rendering-rates"></a>轉譯的費率

混合的實境應用程式是唯一的因為使用者可以自由移動世界中，而且與虛擬的內容，如進行互動，就好像它們是真正的物件。 若要維護這個印象，務必要呈現全像投影，讓他們在世界出現穩定並順利以動畫顯示。 在轉譯[60 每秒畫面格數 (FPS) 的最小](understanding-performance-for-mixed-reality.md)協助達成此目標。 有一些支援反應，高於 60 FPS 的轉譯的混合實境裝置，並為這些裝置強烈建議來提供最佳的使用者體驗更高的反應在呈現。

**深入探討**

要繪製看起來像是全像投影[它們在實際或虛擬世界中穩定](hologram-stability.md)，應用程式需要呈現影像從使用者的位置。 影像轉譯需要時間，因為所有 HoloLens 和其他 Windows Mixed Reality 裝置會預測該使用者的大腦可之映像會顯示在顯示時。 此預測演算法為近似值。 Windows Mixed Reality 演算法和硬體調整所呈現的影像，以說明預測前端的位置和實際的前端位置之間的差異。 此程序可讓使用者看到的出現，如果從正確的位置，轉譯和全像投影認為穩定的映像。 更新最適合小型變更，在前端的位置，以及他們無法完全解釋的一些轉譯之影像的差異，例如影片視差所造成。

**最小的畫面播放速率的 60 FPS 的轉譯，由您進行兩件事，幫助穩定全像投影：**
1. 減少 judder 的外觀，其中的特點在於不均衡的移動和 double 的映像。 更快速的全像移動較低的轉譯的費率與相關聯和更加明顯 judder。 因此，努力，永遠維持 60 FPS （或您的裝置最大轉譯速度） 將有助於避免 judder 移動全像投影。

## <a name="self-motion-and-user-locomotion"></a>自我動作和使用者 locomotion

唯一的限制是您的實體空間; 的大小如果您想要允許非他們可以在其真正的房間往移動的虛擬環境中的使用者，則必須實作一種純虛擬的動作。 不過，不符合使用者的真實的實體動作的持續性虛擬動作通常也會移動嘔吐。 此結果是因為*視覺提示*針對自我動作從*虛擬世界*具有衝突[vestibular 提示](https://en.wikipedia.org/wiki/Vestibular_system)如自我影片來自*真實世界*。

幸運的是，有實作使用者 locomotion 有助於避免發生問題的秘訣：
* 一律讓使用者控制其移動;未預期的自我動作會特別有問題
* 人類是很容易受到重力方向。 因此，非使用者起始的垂直動作特別應該避免。

### <a name="guidance-for-holographic-devices"></a>適用於全像攝影版裝置的指引

若要允許使用者將移至另一個位置，在大型的虛擬環境中的其中一個方法是讓您認為他們要移動的小型物件在場景中。 這個效果可以達成，如下所示：
   1. 提供的介面，使用者可以選取一個位置在虛擬環境中他們想要移動的位置。
   2. 在 選取項目，壓縮到磁碟，以在所要的那個點周圍的場景轉譯。
   3. 同時選取的位置，讓使用者移動它，就好像小型物件。 使用者可以接著移接近其英呎的選取項目。
   4. 取消選取，而在繼續進行轉換整個場景。

### <a name="guidance-for-immersive-devices"></a>沈浸式裝置的指引

先前的全像攝影版的裝置方法無法運作，以及在沉浸式裝置中，因為呈現大型黑色 void 或另一個應用程式需要預設的環境，同時移動 「 磁碟 」。 此處理方式會中斷其中一個的意義的深度。 使用者 locomotion 沈浸式耳機中的一個技巧是 「 閃爍 」 的方法。 這項實作讓使用者控制其動作和提供簡短的印象的移動，但可讓您更簡短的使用者不太可能覺得 disoriented 的純虛擬自我動作：
   1. 提供的介面，使用者可以選取一個位置在虛擬環境中他們想要移動的位置。
   2. 在選取項目，開始並快速淡出轉譯的非常快速模擬 (100 m/s) 動態往該位置。
   3. 完成轉譯之後回到呈現淡出。

## <a name="heads-up-displays"></a>抬頭顯示器

在第一位人員-射擊 videogames 抬頭顯示器 (HUDs) 持續提供資訊，例如播放程式的健全狀況、 迷你地圖和存貨直接在螢幕上。 HUDs 運作，也可將播放程式顯示不含 intruding 遊戲體驗上收到通知。 在混合的實境體驗，HUDs 有可能會造成顯著的 discomfort，而且必須調整以更沈浸式的內容。 具體而言，已嚴格鎖定使用者的前端方向 HUDs 是有可能產生 discomfort。 如果應用程式需要抬頭顯示器，我們建議您*主體*鎖定而不前端的鎖定。 這種處理可以實作為一份顯示可立即轉譯與使用者，但是不會旋轉與使用者的大腦達到臨界值的旋轉。 一旦達到該循環，抬頭顯示器可能會重新調整，來呈現使用者的視野中的資訊。 實作 1:1 抬頭顯示器旋轉和轉譯相對於使用者的大腦動作應該避免。

## <a name="text-legibility"></a>文字可讀性

最佳的文字可讀性有助於減少眼睛疲勞和維護使用者緩和，尤其是在應用程式或要求使用者在 HMD 中讀取的案例。 取決於各種因素，包括各種的顯示屬性 （例如，像素密度、 亮度、 對比）、 功能濾鏡屬性 （例如 chromatic 脫離） 和文字/字型屬性 （例如，特定字型的文字可讀性特性喜歡 weight、 間距、 襯線、 等等的字型、 背景色彩的色彩)。  

一般情況下，我們建議測試特定的應用程式，以利閱讀，並讓字型的大小一樣大，因為是可行的舒適的體驗。 下面我們提供做為起點進行開發的一般指引。 請注意，所有的字型大小以度為單位的報告[視覺化角度](https://en.wikipedia.org/wiki/Visual_angle)而不是特定的實體大小，內含的指引可獲得最佳的全像放置的區域內的任何距離因為它負責的大小文字，而且它會出現在檢視器的距離。 

請參閱[印刷樣式](typography.md)並[Unity 中的文字](text-in-unity.md)頁面如需詳細指導方針。

## <a name="gaze-direction"></a>視線方向

若要避免的眼睛與頸部疲勞內容應該設計，可避免過多的眼睛與頸部移動。
* **避免**視線角度超過 10 度高於 水平分割 （垂直移動）
* **避免**視線角度超過 60 度以下水平分割 （垂直移動）
* **避免**頸部旋轉超過 45 度中央 （水平移動）

最佳 （靜止） 的視線角度視為之間 10-20 度水平，以下標頭通常會以特別是在活動期間，向下稍微傾斜。

## <a name="arm-positions"></a>Arm 位置

當使用者要保留的體驗期間引發手的形狀，就會累積跑車疲勞。 它也可以 fatiguing 為需要使用者重複做出空中點選手勢經過長時間。 因此，建議的經驗，避免需要常數、 重複的筆勢輸入。 可藉由納入短暫休息或供應項目混用筆勢和語音輸入的應用程式互動中達成此目標。
